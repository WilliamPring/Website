GPU Passthrough: Running Windows at Native Speed Inside Linux

Ongoing since ~2022

I run Arch as my host OS full time. Gaming on Linux has improved a lot, but there are still titles and applications that don't work and probably never will. Dual booting was the obvious answer and I did it for a while. It works, but it's genuinely annoying: rebooting constantly to switch, losing access to everything running on the other side, maintaining two separate system setups. GPU passthrough eliminates that tradeoff.

The idea is that Arch runs as the host, and a physical GPU gets passed directly through to a Windows virtual machine. Not emulated. The VM gets actual direct hardware access to the card and performance is essentially native. From inside the VM, Windows sees a real GPU with a real driver. Games run at framerates comparable to a dedicated Windows install.

What You Need

Not every machine can do this. You need a CPU and motherboard that support IOMMU: Intel calls it VT-d, AMD calls it AMD-Vi. Most modern processors include it, but it has to be enabled in the BIOS, and some cheaper boards don't expose the setting even when the CPU supports it.

You also need two GPUs. One stays with the host, one goes to the VM. The most common setup is an integrated GPU or a cheap discrete card for the host and a better discrete card for the guest. My setup uses integrated graphics for the Arch desktop and passes through the dedicated GPU to Windows.

RAM matters more than you might expect. You're pre-allocating memory to the VM and that memory is unavailable to the host while it's running. 16GB is workable if you're careful. 32GB is comfortable.

Enabling IOMMU

Start in the BIOS. Look for VT-d, AMD-Vi, IOMMU, or sometimes just a virtualisation setting in the advanced CPU options. Enable it, save, and reboot.

Then add it to your kernel parameters. Edit /etc/default/grub and append to GRUB_CMDLINE_LINUX_DEFAULT. For AMD:

amd_iommu=on iommu=pt

For Intel:

intel_iommu=on iommu=pt

The iommu=pt flag enables passthrough mode, which tells the IOMMU to only translate for devices that actually need it. This keeps host performance clean for everything that isn't being passed through. Regenerate the GRUB config and reboot:

grub-mkconfig -o /boot/grub/grub.cfg

After rebooting, verify IOMMU is active:

dmesg | grep -e DMAR -e IOMMU

You should see lines confirming IOMMU is enabled and remapping is active.

Checking IOMMU Groups

IOMMU groups are how the hardware isolates devices from each other. For passthrough to work, the GPU you want to pass through needs to be in its own isolated group, or every device in that group needs to be passed through together. You can't split a group.

This script lists all groups and the devices in them:

for g in /sys/kernel/iommu_groups/*; do
  echo "Group ${g##*/}:"
  for d in $g/devices/*; do
    echo -e "\t$(lspci -nns ${d##*/})"
  done
done

Find your GPU in the output. The GPU and its HDMI audio device will typically share one group. If that group contains only those two devices you're in good shape. If unrelated devices are mixed in, you have a problem. Some motherboards handle grouping well, others don't. There's an ACS override patch that can work around bad groupings, but it requires building a custom kernel and has real security implications, so it's worth exhausting every other option first.

Binding the GPU to vfio-pci

Find your GPU's vendor and device IDs:

lspci -nnk

Look for your discrete GPU and its HDMI audio device. Each entry ends with an ID in the format [xxxx:xxxx]. Note both.

Create /etc/modprobe.d/vfio.conf and add both IDs:

options vfio-pci ids=10de:XXXX,10de:XXXX

Then edit /etc/mkinitcpio.conf. In the MODULES array, add the vfio modules before any GPU driver:

MODULES=(vfio_pci vfio vfio_iommu_type1 ...)

Order matters here. vfio_pci must load before amdgpu, nouveau, or nvidia. If the GPU driver loads first it claims the device and vfio can't get it back. Rebuild the initramfs:

mkinitcpio -p linux

After rebooting, confirm the binding took:

lspci -nnk

The GPU should now show Kernel driver in use: vfio-pci instead of an AMD or NVIDIA driver.

Setting Up the VM

Install the required packages:

pacman -S qemu-desktop libvirt virt-manager ovmf

Enable and start libvirt:

systemctl enable --now libvirtd

In virt-manager, create a new VM. The important configuration points, in order:

Set the firmware to UEFI and select the OVMF firmware file. Do not use the default BIOS option; the GPU requires UEFI to initialise properly.

Set the CPU to host-passthrough and enable copy host CPU configuration. This gives the guest access to the actual CPU feature flags, which matters for performance and for some games that check CPU capabilities.

Add a VirtIO SCSI controller before attaching any disk. Attach the disk to the VirtIO SCSI bus. Windows won't recognise this disk during installation without drivers, which you handle in the next step.

Add the passed-through GPU: Add Hardware, PCI Host Device, select your GPU. Add the GPU's HDMI audio device the same way.

Remove the default Spice display and QXL video device. The passed-through GPU handles display now; the virtual display devices will conflict.

For the Windows install, attach the VirtIO driver ISO from Fedora as a second CD-ROM alongside your Windows ISO. When the installer says it can't find a disk, browse to the driver ISO and load the VirtIO SCSI driver. The disk will appear.

CPU Pinning

By default, QEMU schedules VM threads across whatever cores are available. This means the host and the VM compete for the same CPU threads, which causes stuttering and latency spikes inside the VM. CPU pinning assigns specific physical cores exclusively to the VM.

Check your CPU topology first:

lscpu -e

This shows which logical CPUs share a physical core (siblings). For a CPU with 8 cores and hyperthreading, you want to give the VM a matched set: physical cores with both their sibling threads. That way the guest doesn't get half a core.

Edit the VM XML directly:

virsh edit <vm-name>

Add a cputune block with vcpupin entries mapping virtual CPUs to physical CPU threads. Also add an emulatorpin entry to lock the QEMU emulator threads to the host's remaining cores. The exact mapping depends on your CPU topology. The goal is zero overlap between host cores and guest cores.

Memory Hugepages

Normal memory pages are 4KB. The VM constantly translates virtual to physical addresses across all of them. Hugepages bump that to 2MB per page, which reduces the number of translations significantly and improves memory latency inside the guest.

Add this to your kernel parameters in /etc/default/grub:

hugepages=2048

That pre-allocates 2048 pages at 2MB each, totalling 4GB. Size it to roughly match the RAM you're assigning to the VM. Static hugepages are claimed at boot before memory fragments, so they're more reliable than dynamic allocation at runtime.

In the VM XML, tell libvirt to use them:

<memoryBacking>
  <hugepages/>
</memoryBacking>

CPU Governor

Linux's default CPU governor throttles clock speed when load is low. When the VM is running you want the CPUs pegged at maximum frequency. You can automate the switch using libvirt hooks.

Create a script at /etc/libvirt/hooks/qemu. The hook receives the VM name, the operation (start, stopped), and the phase as arguments. On start, set the governor to performance for every CPU. On stop, restore it to your preferred idle governor. Make the file executable.

This is one of the more noticeable tuning steps. Switching from schedutil to performance removes a source of latency that shows up as microstutters in games, particularly in CPU-heavy scenes.

Audio

Getting audio out of the Windows VM and into the host is its own problem. The approach I use is Scream, a virtual audio driver for Windows that streams audio over the local network (or via shared memory on the same machine).

Install the Scream driver in Windows. On the Linux side, run the Scream receiver, which outputs to whatever PipeWire sink you point it at. From there the audio goes through EasyEffects the same as any other source on the system. Latency over shared memory is low enough that it's not noticeable in practice.

This is part of why the PipeWire and Jack patchbay setup in my Linux machine write-up is more elaborate than it might seem. Routing VM audio through the same patchbay as everything else means it behaves consistently with the rest of the system.

Windows 11

Windows 11 requires Secure Boot and TPM 2.0. Both are achievable in a VM.

For Secure Boot, OVMF includes Secure Boot support. In the VM firmware settings, enable it and enroll the Microsoft keys. virt-manager exposes this in the firmware selection when you pick the Secure Boot variant of the OVMF firmware file.

For TPM 2.0, install swtpm:

pacman -S swtpm

Add a TPM device to the VM via virt-manager: Add Hardware, TPM, set model to TIS and version to 2.0 with the emulated backend. libvirt handles starting and stopping swtpm automatically when the VM runs.

Some games with anti-cheat check for hypervisor signatures and refuse to run if they detect a VM. You can mask this from the guest by adding to the VM XML:

<features>
  <hyperv>
    <vendor_id state="on" value="randomstring"/>
  </hyperv>
  <kvm>
    <hidden state="on"/>
  </kvm>
</features>

This tells QEMU not to expose KVM to the guest. The performance impact is minimal. The vendor ID value can be any alphanumeric string up to 12 characters.

Common Issues

If the screen freezes at boot on the host, the disk encryption passphrase prompt is probably appearing before the display initialises. The system is waiting for input. Type the passphrase blind and hit enter. It goes through.

If the GPU fails to reset cleanly between VM starts, you may be hitting the GPU reset bug. This affects certain AMD GPU families where the card doesn't fully reset when the VM shuts down, leaving it in a broken state. The vendor-reset module from AUR implements proper reset support for affected cards. Install it and load it before the vfio modules.

If you get a black screen when the VM starts, check that vfio-pci actually claimed the GPU. Run lspci -nnk and look at the GPU entry. If it shows amdgpu or nvidia as the kernel driver instead of vfio-pci, the GPU driver loaded first and won the race. Double-check the module ordering in mkinitcpio.conf and rebuild the initramfs.

If performance inside the VM feels uneven even after pinning and hugepages, check whether the CPU governor is actually switching. cat /sys/devices/system/cpu/cpu0/cpufreq/scaling_governor should return performance while the VM is running. If it doesn't, the hook isn't firing.
